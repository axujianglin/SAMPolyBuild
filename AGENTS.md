# AGENTS.md

## 一、项目目标

- 当前项目基于 SAMPolyBuild。
- 项目目标是实现遥感建筑轮廓交互提取。
- 后续将增加单点自适应 bbox、正负点交互修正、建筑轮廓规则化、GIS 导出等功能。

## 二、开发原则

- 不修改 SAMPoly 模型结构。
- 不重新训练模型。
- 优先采用最小修改方案。
- 必须保留原有手动 bbox 功能。
- 新增功能必须向后兼容，不破坏现有推理、测试和训练入口。
- 不允许大范围重构无关代码。
- 修改前应先阅读相关调用链，确认现有数据结构和推理流程。
- 优先复用现有函数、配置和脚本，避免重复实现相同逻辑。

## 三、Git 规范

- 禁止直接在 main/master 分支上开发。
- 每个功能必须新建 feature 分支。
- 修改前必须检查 `git status`。
- 修改后必须检查 `git diff`。
- 每个阶段完成后必须提交 commit。
- commit 信息必须使用 `feat`、`fix`、`refactor`、`docs`、`test` 前缀。
- 不得回滚或覆盖他人未确认的修改。

## 四、代码规范

- 函数职责单一，一个函数只处理一个明确任务。
- 不写死路径，路径应通过参数、配置文件或集中配置传入。
- 不使用难以理解的魔法数字；必要常量必须命名，并说明含义。
- 配置参数集中管理，避免散落在多个脚本中。
- 失败时必须返回或抛出明确错误信息，不允许静默失败。
- 新增交互逻辑应与核心推理逻辑解耦，便于无 GUI 测试。
- 新增代码应保持与现有项目风格一致，避免引入无关依赖。

## 五、测试规范

- 新增功能必须提供独立测试脚本。
- 测试脚本不能依赖 GUI。
- Debug 输出必须可开关。
- 不允许只做人工测试。
- 测试应覆盖正常输入、缺失输入和错误输入。
- 涉及 polygon 输出的功能，应验证输出坐标系、点数有效性和保存结果格式。

## 六、文档规范

- 新增功能必须更新 README。
- README 需要说明使用方式、参数、限制和调试输出。
- 新增脚本需要说明输入、输出、依赖模型权重和示例命令。
- 行为变化必须在文档中明确标注，避免影响原有使用方式。

## 七、Runtime Environment

- 当前项目源码位于 Windows。
- 实际运行环境位于 Linux 远程服务器。
- Linux 运行环境使用 Conda environment：`sampoly`。
- Python、PyTorch、CUDA、SAMPoly 相关依赖均安装在 Linux 环境中。
- Codex 默认不要直接执行 Python 测试。
- Codex 默认不要假设本机具备项目运行环境。
- 当需要测试时，Codex 应生成测试命令，等待用户在 Linux 环境执行，再根据用户返回日志分析问题。
- 所有测试步骤必须同时提供 Linux 执行命令、预期输出和调试建议。
